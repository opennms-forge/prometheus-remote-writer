= Installation

== Compatibility

[cols="1,3"]
|===
| OpenNMS Horizon  | {horizon-min-version}+
| JVM              | Temurin / OpenJDK {java-target-version} (matches the Horizon container)
| Apache Karaf     | {karaf-version}
| Integration API  | `opennms-integration-api` v2.0
|===

The plugin's OSGi bundle is compiled to Java {java-target-version} bytecode
to match Horizon's runtime; running on an older JVM fails feature
resolution at install time.

== Install on a running Horizon instance

The plugin ships as a Karaf KAR (`prometheus-remote-writer-kar-X.Y.Z.kar`).
Two install paths are available — pick whichever fits your deployment.

=== Path A: KAR drop (current shipping path)

[source,bash]
----
# 1. Download the KAR from the GitHub Release matching your installed version.
#    Example for v0.3.0:
curl -L -o /opt/opennms/deploy/prometheus-remote-writer.kar \
    https://github.com/OpenNMS/prometheus-remote-writer/releases/download/v0.3.0/prometheus-remote-writer-kar-0.3.0.kar
----

Karaf's `deploy` directory hot-installs `.kar` artifacts on drop. Confirm
from the Karaf shell:

[source,bash]
----
ssh -p 8101 admin@localhost
karaf@root()> bundle:list -s | grep prometheus-remote-writer
----

=== Path B: feature install via Maven coordinates (deferred)

The Maven-coordinate install (`feature:repo-add mvn:…/xml/features`)
requires the project's artifacts to be published to a Maven repository.
That step is deferred until the repo migrates under the OpenNMS namespace
and a publish target is chosen — see `RELEASING.md` in the source tree.

== Activate the plugin as the active TSS

Set the time-series strategy in `etc/opennms.properties.d/timeseries.properties`:

[source,properties,subs=attributes+]
----
{tss-strategy-key} = {tss-strategy-value}
----

Restart OpenNMS for the strategy switch to take effect. The next collector
flush sends samples through the plugin.

== Minimum configuration

Drop the bare minimum at `{cfg-file}`:

[source,properties]
----
write.url = https://mimir.example.com/api/v1/push
read.url  = https://mimir.example.com/prometheus
----

`read.url` is the backend's *Prometheus-compatible root*. The plugin
appends `/api/v1/series` and `/api/v1/query_range` itself — do *not*
include `/api/v1` in the configured value. See
<<configuration,Configuration reference>> for the full list of knobs and
backend-specific URL shapes.

== Verify

From the Karaf shell:

[source]
----
karaf@root()> bundle:list -s | grep prometheus-remote-writer
karaf@root()> opennms:prometheus-writer-stats
----

`opennms:prometheus-writer-stats` prints all plugin counters and gauges.
Watch `samples_written_total` tick up as OpenNMS pushes its first samples
through the plugin.

[NOTE]
.Default labels need OpenNMS metatag config to be useful
====
By default, samples arriving at the plugin carry only `name` and
`resourceId` — OpenNMS does not attach node/interface/asset metatags
unless you configure them. See <<label-mapping>> for the four-line
metatags config that turns those tags on.
====

== Local sandbox (e2e)

The repository's `e2e/` directory contains a Docker Compose stack that
stands up an OpenNMS Horizon core, Grafana, and one Prometheus-compatible
backend of your choice. It is the quickest way to see the plugin work
end-to-end on a laptop.

Prerequisites: Docker 24+ with Compose v2.

The fastest way to prove the plugin works is `make smoke`:

[source,bash]
----
make smoke                               # all backends, sequential
make smoke BACKENDS=prometheus           # single backend
make smoke TIMEOUT=900 BACKENDS=mimir    # bump the per-backend deadline
----

For interactive exploration, see `e2e/README.md` in the source tree.
